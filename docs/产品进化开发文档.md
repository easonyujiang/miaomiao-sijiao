## 一、审计摘要

### 1.1 已确认的核心价值

对话中反复出现的三个支柱：

1. **学习结果可感知** —— 用户必须知道「我学到了什么、能产出什么」。
2. **长尾选择权** —— 学习结束后，系统可以提供延续，但用户必须能拒绝。
3. **可复盘** —— 学习不是一次性消费，而是可以回看、巩固、再生产。

这三个点共同指向一个产品定位：

> **不是内容消费工具，而是「学习结果生产工具」。**

### 1.2 已放弃的探索方向

| 方向 | 放弃原因 | 审计意见 |
|---|---|---|
| 抖音流量娱乐 | 与「严肃学习」定位冲突 | ✅ 正确放弃，避免品牌稀释 |
| 收藏夹学习 | 过于泛化，缺少闭环 | ⚠️ 有价值，但应降级为「复习入口」而非主路径 |

### 1.3 当前代码资产盘点

| 妙喵桌宠 | ✅ 就绪（含语音/朗读/彩蛋） | 情感化学习陪伴、报告交付仪式 |

## 二、产品进化方向：从「私教」到「学习结果银行」

### 2.1 一句话定位

> **妙喵私教 = 用 10 分钟视频，帮你生产一份「可带走、可复盘、可拒绝延续」的学习报告。**

### 2.2 核心循环

```
观看视频 → 互动问答 → 知识点固化 → 生成报告 → 选择长尾（接受/拒绝） → 进入复盘队列
```

### 2.3 与竞品差异

| 竞品/替代方案 | 它们做什么 | 妙喵做什么 |
|---|---|---|
| 抖音知识短视频 | 娱乐化被动消费 | 结构化主动生产 |
| Notion/飞书笔记 | 自己记笔记 | AI 帮你提炼 + 生成可交付物 |
| 传统网课 | 完整课程体系 | 碎片输入 → 结构化输出 |
| Anki | 卡片复习 | 基于真实视频内容的语境化复习 |

## 三、学习报告系统（MVP 核心）

### 3.1 报告内容结构

每完成一段学习，生成一份「妙喵学习报告」：

```markdown
# 学习报告 · 《视频标题》

## 1. 我学到了什么
- 核心概念 1：一句话定义
- 核心概念 2：一句话定义
- 关键金句：原文引用 + 妙喵解读

## 2. 我能产出什么
- 应用题/练习（1-2 道）
- 可发朋友圈/小红书的知识点卡片
- 可写进工作文档的论点/模板

## 3. 我的掌握度
- 自评星级 ⭐⭐⭐☆☆
- 妙喵评估：基于对话问答的正确率
- 薄弱点：XXXX（建议复习）

## 4. 下一步建议
- 接受：继续学习《相关视频 A》
- 拒绝：把本报告加入复盘队列
```

### 3.2 生成时机

| 触发条件 | 报告类型 |
|---|---|
| 用户主动说「生成报告」 | 完整版 |
| 视频观看到 90% + 完成 ≥2 轮对话 | 自动简化版 |
| 用户完成应用题 | 增强版（含产出物） |

### 3.3 前端页面

- `/reports` —— 报告列表
- `/reports/[id]` —— 报告详情
- `/reports/[id]/card` —— 知识点卡片（可分享）
- 桌宠交付：生成报告时播放 `celebrating` + `level-up` + 撒花

## 四、长尾效应机制（尊重用户选择权）

### 4.1 设计原则

> 不是「推荐更多视频」，而是「提供可选的下一步」。

### 4.2 长尾选项

学习报告末尾提供 3 个按钮：

1. **深入一点** —— 同一主题的进阶视频/片段
2. **练一练** —— 基于本内容的应用题
3. **先放着** —— 加入复盘队列，7 天后提醒

### 4.3 拒绝的处理

- 用户点击「先放着」→ 加入 `review_queue`
- 用户关闭推荐 → 记录偏好，下次减少同类推荐
- 用户明确说「不需要」→ 触发 `moodFor('不需要')` 进入 `failed` 安慰态，避免打扰

### 4.4 后端数据模型

```sql
CREATE TABLE review_queue (
    id TEXT PRIMARY KEY,
    visitor_id TEXT NOT NULL,
    video_id TEXT NOT NULL,
    report_id TEXT,
    scheduled_at INTEGER NOT NULL,  -- 复盘提醒时间
    status TEXT DEFAULT 'pending',  -- pending / done / dismissed
    created_at INTEGER DEFAULT (strftime('%s', 'now'))
);
```

## 五、复盘机制

### 5.1 复盘周期

| 阶段 | 时间 | 形式 |
|---|---|---|
| 初次复盘 | 学习后 1 天 | 推送 3 道快问快答 |
| 二次复盘 | 学习后 7 天 | 让用户用自己的话复述知识点 |
| 三次复盘 | 学习后 30 天 | 应用题/场景题 |

### 5.2 复盘触发

- Chrome 插件推送通知
- 邮件/飞书机器人（可选）
- 用户进入 `/diary` 时显示「今日复盘」

### 5.3 复盘效果可视化

`/learning-diary` 页面增加：

- 学习热力图
- 知识点掌握度曲线
- 已生成报告数 / 已复盘数
- 妙喵成长值（把学习量转化为宠物成长）

## 六、商业模式（创意 + 实用）

### 6.1 免费层

- 每月 5 份学习报告
- 基础复盘提醒
- 标准妙喵形象

### 6.2 付费层（Pro）

| 功能 | 价值点 |
|---|---|
| 无限报告 | 高频学习者刚需 |
| 高级报告模板 | 工作汇报/朋友圈/小红书卡片 |
| 自定义妙喵皮肤 | 情感付费 |
| 专属复盘计划 | 备考/考证人群 |
| 团队/企业版 | 培训内容自动沉淀 |

### 6.3 B 端延伸

> **企业知识库沉淀**：把内部培训视频变成结构化学习报告 + 可搜索知识点库。

这与当前 `extension/` + `website/` 双端架构天然匹配。

## 七、技术实现路线图

### Phase 1：学习报告 MVP（2 周）

1. 后端：新增 `reports` 表 + `/api/reports` CRUD
2. 后端：LLM prompt 生成报告（基于视频 transcript + 对话记录）
3. 前端：`/reports` 列表页 + `/reports/[id]` 详情页
4. 桌宠：报告生成时触发庆祝动画 + 音效

### Phase 2：复盘与长尾（2 周）

1. 后端：`review_queue` 表 + 定时任务
2. Chrome 插件：复盘通知
3. 前端：`/diary` 增加复盘入口和学习统计
4. 长尾推荐：基于视频标签/知识图谱的简单相似推荐

### Phase 3：商业化与 B 端（4 周）

1. 报告模板系统
2. 用户订阅/支付
3. 企业版：多站点部署 + 团队管理
4. 数据看板：学习转化率、留存、报告完成率

## 九、风险与取舍

| 风险 | 应对方案 |
|---|---|
| 报告生成质量不稳定 | 先用规则 + LLM 混合，关键字段人工校验 |
| 用户不点「生成报告」 | 自动简化版报告 + 强提示 |
| 复盘打扰用户 | 默认关闭，用户主动开启；拒绝后降级频率 |
| 与抖音定位冲突 | 完全不碰公域流量，专注私域/工具属性 |
| 收藏夹功能被放弃 | 可保留为「从收藏链接生成学习报告」的入口 |
